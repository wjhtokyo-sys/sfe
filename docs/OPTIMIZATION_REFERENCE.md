# SFE 优化参考（仅方案评估，不改代码与数据）

> 目标：在**完全保证现有功能**前提下，对 SFE 的业务链路与数据组织进行压缩与简化，供后续迭代参考。
> 范围：本文件仅为优化参考，不包含任何实施改动。

## 1. 结论摘要

在不牺牲现有功能前提下，SFE 有明显可压缩空间：

- 业务链路复杂度可降低约 **25%~40%**
- 数据模型认知负担可降低约 **30%**
- 重复逻辑与边缘状态可降低约 **35%~50%**

风险分级建议：

- **低风险（优先）**：命名统一、只读语义层、规则文档收敛、状态一致性校验
- **中风险（分阶段）**：状态机收敛、FIFO pending 规则标准化
- **高风险（谨慎）**：核心表深度合并与迁移

---

## 2. 现状问题（可压缩来源）

当前主链路：

`采购入库 -> FIFO分配 -> 挂起任务人工处理 -> 账单生成 -> 状态推进`

流程方向正确，但表达方式偏分散：

1. bill 存在 `status/payment_status/shipping_status` 多状态并行，维护成本高
2. `allocated_by` 使用字符串编码来源，前后端解析成本高
3. `fifo_pending_tasks` 承载多类场景，规则边界易膨胀
4. 局部 API 路径存在历史并行（如客户相关路径）
5. 模型定义存在历史痕迹，增加理解成本

---

## 3. 优化建议（不减功能）

### 3.1 业务链路压缩

#### A. 统一“入库分配结果”语义层（低~中风险）
- 将自动分配/人工匹配/无匹配转客户统一映射为一致输出：
  - `source_type`（auto/manual/no_match_assign）
  - `source_ref`（PO号）
- 减少对 `allocated_by` 文本解析的直接依赖

**收益**：前后端判断分支明显减少；底层表可保持不动。

#### B. 账单状态收敛为“主状态+派生状态”（中风险）
- 保持兼容期内字段不删除
- 引入唯一主状态（stage）
- 其他状态做派生或强一致性约束

**收益**：减少非法状态组合，提升追踪与维护性。

#### C. FIFO pending 规则标准化（中风险）
- 对 `reason_code` 建立固定字典：
  - 触发条件
  - 可执行动作
  - 动作后数据去向
  - 是否允许部分处理

**收益**：防止原因码和处理分支无序扩展。

### 3.2 数据组织压缩

#### D. 软压缩优先：建立只读语义层（低风险）
建议建立统一查询/视图（先不改表）：

1. `v_arrival_allocation`：到货与分配统一观察面
2. `v_bill_pipeline`：账单阶段统一观察面
3. `v_fifo_pending_actionable`：待处理任务可操作观察面

**收益**：先降复杂度，再考虑结构迁移。

#### E. 快照策略固化（低风险）
- 明确哪些业务必须 snapshot（如 jan/item_name）
- 明确哪些业务可引用主数据

**收益**：防止历史单据被主数据变更污染。

#### F. 主数据接口入口统一（低风险）
- 客户/商品/供应商逐步收敛到同一接口风格
- 保留旧路径作为兼容层

**收益**：降低 API 认知与权限边界复杂度。

---

## 4. 明确不建议直接做的高风险动作

1. 不建议直接合并 `customer_orders / allocations / inventory_lots`
   - 三者分别代表需求、库存、分配事实，强行合并会破坏可追溯性
2. 不建议去掉快照字段
   - 历史单据会随主数据变动而失真
3. 不建议立即删除旧状态字段
   - 应先经历双轨一致性校验期

---

## 5. 分阶段落地路线（供后续立项）

### Phase 0（低风险，1~2周）
- 统一术语与规则文档
- 增加状态一致性校验
- 建立只读语义层（查询/视图）

### Phase 1（中风险，2~4周）
- API 语义统一输出
- pending 规则收敛
- allocated_by 结构化映射

### Phase 2（中~高风险，按需）
- 字段级精简（在依赖确认后）
- 可选结构迁移（带回滚策略）

---

## 6. 验收原则（保证“功能不损失”）

- 所有现有业务流程可完整走通
- 数据可追溯性不下降
- 历史单据结果不变化
- 权限边界不变化
- 关键报表口径不变化
