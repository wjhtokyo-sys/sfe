# SFE 业务与数据结构统一规范（最终版）

> 本文档为 SFE 项目唯一业务语义与流程约束说明。
> 目标：统一概念、消除歧义、固化流程与状态守卫，作为后续开发与审计基线。

## 一、专用名词统一定义（本系统唯一解释）

### 商品（Item）
- 静态商品模板信息，用于展示与下单入口。
- 不代表库存实物，不参与库存计算。
- 不可物理删除（仅可下架）。

### 货品（Inventory Goods / Lot Quantity Unit）
- 实际到货并进入库存流转的实物数量单位（点数）。
- 必须归属某一个商品。
- 参与 FIFO 与库存变化。
- 客户不可见。

### 货位（Storage Location）
- 货品在仓库中的物理存放位置。
- 属于货品属性，仅后台可见。
- 当前不做多仓复杂逻辑。

### 订单（Customer Demand）
- 客户在商品页面发起的需求请求，属于“未到货需求池”。
- 不占库存，不产生财务。
- 允许部分履约。
- 客户仅可见自身订单。

### 订单表（customer_orders）
- 存储客户需求记录的表。
- 一行即一条需求（无订单头）。
- 仅表达“要多少”，不保证一定供货成功。

### 客户履约分配（Allocation）
- 后台将已到货库存（FIFO 批次/货品批次）锁定并分配给某客户需求。
- 本质为库存预占用。
- 客户不可见。

### 客户履约分配表（allocations）
- 记录每一笔“库存批次 -> 客户需求”的锁定关系（数量、状态、操作者）。
- 是库存与销售之间的桥梁层。
- 原“到货单”统一技术语义为“客户履约分配”。

### 账单（Bill）
- 已确定实际销售价格、允许客户付款的交易单元。
- 是客户可支付的最小单位。

### 账单表（bills + bill_lines）
- 客户可见的正式结算数据（账单头 + 账单行）。
- 包含销售单价与金额。
- 驱动付款/收款/发货/收货状态流转。

### FIFO 挂起（FIFO Pending）
- 库存可满足多个客户需求但无法自动唯一分配时的待裁决状态。
- 仅超级管理员可处理与调整分配。

### 客户（Customer）
- 外部买方用户。
- 仅可见自身订单、账单及与自己相关的物流轨迹。
- 不可见库存、成本、货位、分配细节。

### 销售管理员（Sales Admin）
- 负责日常审核与单据生成（尤其账单生成、付款审核）。
- 不能干预 FIFO 挂起裁决。
- 不能修改库存批次分配核心。

### 超级管理员（Super Admin）
- 最高权限。
- 负责 FIFO 挂起裁决、关键状态推进、数据修正与权限管理。
- 可查看审计日志。

## 二、整体数据流（唯一正确路径）

客户下单（订单表）
-> 实际到货形成货品并进入 FIFO 库存
-> 后台执行客户履约分配（锁库存：批次 -> 需求）
-> 多条分配合并生成账单（在账单行赋予实际销售价格）
-> 客户付款声明
-> 后台确认收款
-> 后台发货
-> 客户确认收货
-> 归档

**硬约束：**
- 任何流程不得跳层或逆向。
- 任何金额只在账单层出现。

## 三、数据层结构（收敛版）

> 统一规则：本系统采用“行级订单”，无订单头；移除语义歧义字段。

### 1) 商品信息表：`items`

字段：
- `id`
- `jan`（唯一）
- `brand`
- `name`
- `spec`（可选）
- `msrp_price`（厂家建议零售价，仅展示参考）
- `in_qty`（入数，默认 1，仅展示/参考，不参与库存计算）
- `is_active`
- `created_at`
- `updated_at`

规则：
- 商品禁止物理删除；被货品引用后仅允许下架。
- 商品价格不参与库存或财务计算。

可见性：
- 客户：可读公开字段。
- 销售管理员：可修改 `msrp_price`。
- 超级管理员：全功能。

### 2) 订单表：`customer_orders`（行级订单，无订单头）

字段：
- `id`（即 `order_line_id`）
- `customer_id`
- `item_id`
- `jan_snapshot`
- `item_name_snapshot`
- `qty_requested`
- `qty_allocated`（累计已分配数量）
- `status`（`open` / `closed` / `cancelled`）
- `created_at`

规则：
- 订单不占库存、不含金额。
- 允许部分履约（`qty_allocated < qty_requested`）。
- 关闭条件：需求已全部分配或被取消。
- 订单永远不直接连接库存批次（必须经 allocations 桥接）。

可见性：
- 客户仅看自己的；销售管理员看全部；超级管理员全权限。

### 3) FIFO 库存层：`inventory_lots` + `stock_moves`

核心约束：
- 库存唯一真实来源是 `lot.qty_remaining`。
- 任何销售过程不得直接改库存，必须通过“分配/出库”服务产生流水。
- FIFO 顺序入库后不可被普通角色修改。
- 挂起裁决仅超级管理员可处理。

可见性：
- 客户不可见；销售管理员只读；超级管理员可处理挂起。

### 4) 客户履约分配表：`allocations`

字段：
- `id`
- `customer_id`
- `order_line_id`
- `lot_id`
- `item_id`
- `qty_allocated`
- `fifo_rank_snapshot`
- `status`（`active` / `released` / `billed`）
- `allocated_by`（操作者标识）
- `created_at`

规则（必须遵守）：
- `allocations` 是库存锁定唯一来源，禁止旁路锁定。
- 分配成功后，对应库存视为已占用但未出库（业务锁定）。
- FIFO 挂起裁决仅超级管理员可操作。
- 进入 `billed` 后不允许改数量。
- 纠错必须走“释放/冲销（released）-> 重新分配”流程。

可见性：
- 客户不可见；销售管理员只读；超级管理员全功能。

### 5) 账单表：`bills` + `bill_lines`

账单头 `bills` 字段：
- `id`
- `customer_id`
- `bill_no`（唯一）
- `status`（`draft` / `issued` / `archived`）
- `payment_status`（`unpaid` / `paid` / `received`）
- `shipping_status`（`not_shipped` / `shipped` / `delivered`）
- `total_amount`
- `currency`
- `payment_confirmed_at`（可选）
- `created_at`

账单行 `bill_lines` 字段：
- `id`
- `bill_id`
- `allocation_id`
- `item_id`
- `jan_snapshot`
- `item_name_snapshot`
- `qty`
- `sale_unit_price`（实际销售价格）
- `line_amount`
- `created_at`

规则：
- 仅可由 allocations 生成账单行。
- 账单行生成时确定并锁定 `sale_unit_price`。
- 客户仅可见账单层价格与金额。
- 账单不允许物理删除；仅未付款可作废，其余只能归档。
- 账单状态必须走 State Guard。

可见性：
- 客户：查看自己的账单，可付款声明/收货确认。
- 销售管理员：生成账单、审核付款。
- 超级管理员：确认收款、推进发货、归档等关键状态。

## 四、State Guard（状态守卫）统一规则（生产硬约束）

所有状态字段必须满足：
1. 只允许正向流转，不允许回退。
2. 禁止接口直接更新状态字段。
3. 所有状态变更必须通过统一状态守卫校验。
4. 关键状态变化必须写入审计日志。
5. 任一环节失败必须保持“库存、分配、账单”三者一致。

---

## 五、必须长期盯住的风险点（最终版）

1. `allocations` 必须是唯一库存锁定来源（禁止旁路）。
2. FIFO 挂起仅超级管理员可裁决。
3. 订单与库存永不直接关联（必须经 allocations）。
4. 账单生成必须基于 allocations，而非订单。
5. 所有金额只存在账单层（商品仅 MSRP 展示参考）。

## 七、前端交互技术要求（防重复提交）

1. 全项目所有业务按钮（`click-btn`）必须启用双击预防。
2. 单次点击后按钮进入短时锁定（建议 800ms），锁定期间拒绝重复触发。
3. 该策略适用于新增、修改、删除、状态流转、导入、生成账单等关键动作。
4. 禁止仅依赖后端幂等来兜底，前端必须先行防抖/防重入。
5. 后续新增按钮如未加 `click-btn`，需在评审中说明理由并补等效防重复机制。
